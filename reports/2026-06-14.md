### 1. Trip report: June 2026 ISO C++ standards meeting (Brno, Czechia) [🔗](https://herbsutter.com/2026/06/13/brno-trip-report/)
> **来源**：Herb Sutter

- **一句话总结**：C++29 首次标准会议在捷克布尔诺结束，通过了 19 项核心语言提案和 18 项标准库提案，并推进了内存安全 Profiles 和系统性 UB 治理的路线图。
- **核心要点**：
  - **P3596R3 "Undefined Behavior and IFNDR Annexes"** — 历时数年，由 Joshua Berne、Timur Doumler、Shafik Yaghmour 等人完成了 C++ 所有未定义行为的完整目录编制。这是首次将标准中所有 UB 情形系统性地编入两个 Annex，为下一步逐个消除 UB 奠定基础。接下来的半年将逐项审查并力争在 C++29 中落地。
  - **P3097R3 "Contracts for C++: Virtual functions"** — C++26 Contracts 发布不到 3 个月，虚函数契约支持即通过。重写函数的前置/后置条件与基类独立，虚函数调用时会同时检查静态类型和最终覆盖者的断言。
  - **P3668R4 "Defaulting Postfix Increment and Decrement Operations"** — `operator++(int) = default` 现在合法，编译器自动生成委托给前缀版本的规范实现，消除手写样板代码。
  - **P2287R6 "Designated-initializers for Base Classes"** — 支持 `B{.a=1, .b=2}` 形式的基类指定初始化，直接初始化继承成员。
  - **P3091 "Better Lookups for map, unordered_map, and flat_map"** — 新增 Python 风格的 `.lookup(key)` 方法，返回 `std::optional`，避免 `operator[]` 的无意插入问题。
  - **P3125R6 "constexpr pointer tagging"** — Hana Dusíková 的 `pointer_tag_pair<Pointer,Tag>` 类型，提供可移植的指针低位比特标记支持。
  - **P3248 "Require [u]intptr_t"** — `intptr_t`/`uintptr_t` 从可选变为必需，方便底层代码编写。
  - **内存安全进展**：语言演进分组（EWG）批准夏季/秋季逐行审查 P3100（系统性解决 UB 的框架），SG23 决定制定 Profiles 规范以提供静态分析和子集化规则，目标纳入 C++29。

### 2. Orthodox C++ (2016) [🔗](https://bkaradzic.github.io/posts/orthodoxc++/)
> **来源**：Branimir Karadžić (Hacker News, 81 points)

- **一句话总结**：一份倡导"最小子集 C++"（又称 C+）的经典指南，主张在 C 的基础上只引入确有必要的 C++ 特性，避免 Modern C++ 的过度复杂性。
- **核心要点**：
  - **核心哲学**："Within C++, there is a much smaller and cleaner language struggling to get out." — Bjarne Stroustrup。Orthodox C++ 就是尝试提取那个更小的语言。
  - **不要用异常**：异常处理是唯一需要复杂运行时支持的 C++ 特性，即使不用也有隐性成本（优化器受限、每个对象构造析构隐藏代码）。
  - **不要用 RTTI**：运行时类型识别带来不必要的耦合和性能开销。
  - **不要用 iostream**：推荐 printf 风格代替 `<iostream>` / `<stringstream>`，使用 C 头文件（`<stdio.h>` 而非 `<cstdio>`）。
  - **谨慎使用 STL 分配器**：除非明确关心内存管理，否则避免 STL 中涉及内存分配的部分。
  - **模板元编程适度使用**：只在确实能降低代码复杂度时使用，避免"学术式自渎"。
  - **等待特性成熟**：对新标准特性保持观望态度，比如 `constexpr` 在 C++11 中几乎不可用，到 C++14 才真正可用。
  - **不推荐 Modules**：列举了 Modules 带来的编译器兼容性、构建系统复杂度、调试困难等缺点。
  - **现实意义**：2026 年回看，Orthodox C++ 的思想对许多嵌入式、游戏引擎、性能敏感场景仍有参考价值；C++26/29 的 Contracts、Reflection 等新特性也为传统拒绝 Modern C++ 的团队提供了重新评估的契机。

### 3. s3cpp: Lightweight C++23 S3 client with no extra deps [🔗](https://github.com/ggcr/s3cpp)
> **来源**：Hacker News (Show HN)

- **一句话总结**：一个零额外依赖（仅 libcurl + OpenSSL）的 C++23 AWS S3 客户端库，展示了现代 C++ 在实际网络编程中的完整应用。
- **核心要点**：
  - **技术栈**：C++23，CMake 构建，GoogleTest 单元测试（63 个测试用例）。
  - **架构模块化**：分为 `httpclient`（HTTP 传输）、`auth`（AWS Signature V4 签名协议）、`xml`（自定义 FSM XML 解析器）。
  - **API 设计**：使用 C++23 `std::expected` 进行错误处理，`std::println` 输出，支持 CreateBucket、ListBuckets、ListObjects（含 Paginator 自动处理 continuation token）、HeadBucket、DeleteObject/DeleteBucket 等完整 S3 操作。
  - **性能优势**：基准测试显示在连续小任务场景下比 AWS Go SDK 和 AWS C++ SDK 有明显性能提升。
  - **本地测试**：支持 Docker MinIO 容器进行本地集成测试。
  - **注意**：作者标注为业余项目（not production-ready），但代码设计和 C++23 特性运用值得学习参考。

### 4. What science can tell us about C and C++'s security (2020) [🔗](https://alexgaynor.net/2020/may/27/science-on-memory-unsafety-and-security/)
> **来源**：Alex Gaynor (Hacker News)

- **一句话总结**：跨 Chrome、Android、iOS/macOS、Microsoft、Firefox、Ubuntu 内核的大规模实证数据一致表明：C/C++ 代码库中超过 65%（某些场景高达 90%）的安全漏洞源自内存不安全。这是编程语言领域最稳健的实证结论之一。
- **核心要点**：
  - **数据全景**：Android Chrome（>65% High & Critical bugs 是内存安全）、iOS 12（66.3%）、macOS Mojave（71.5%）、Chromium（~70%）、Microsoft 年度 CVE（~70%）、Firefox CSS 子系统（73.9%）、Ubuntu Linux 内核（65%）。
  - **已被利用的 0-day 漏洞也符合这一趋势**，说明这不是理论问题而是实际威胁。
  - **逻辑推导**：如果将 Venn 图分为三类漏洞——(1) 仅内存不安全语言才有、(2) 任何语言都可能存在、(3) 仅内存安全语言才有——那么切换到内存安全语言可以消灭至少 65% 的漏洞，而第(3)类（如 unsafe deserialization）的数量级远小于第(1)类。
  - **与 C++29 的联系**：这正是 ISO C++ 委员会在 Brno 会议上积极推进 UB 治理和内存安全 Profiles 的根本原因——用工程手段系统性解决这一被反复验证的安全问题。
  - **结论**：作者认为"用内存安全语言重写大型 C++ 项目可以减少~70% 安全漏洞"是目前编程语言领域最被低估的实证结论。
