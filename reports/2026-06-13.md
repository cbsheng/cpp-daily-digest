### 1. C++26: Structured Bindings can introduce a Pack [🔗](https://isocpp.org//blog/2026/06/cpp26-structured-bindings-can-introduce-a-pack-sandor-dargo)
> **来源**：isocpp.org (Sandor Dargo)

- **一句话总结**：C++26 允许结构化绑定引入参数包（pack），大幅简化 tuple 和结构体的分解操作，消除嵌套 `std::apply` 的模板元编程样板代码。
- **核心要点**：
  - 支持 `auto [...xs] = f()` 将整个 tuple 分解为一个 pack，`auto [x, ...rest] = f()` 捕获尾部剩余元素，`auto [x, ...rest, z]` 支持同时命名首尾元素
  - 多个 pack 在同一绑定中非法（`auto [...a, ...b]` 为 ill-formed），但支持通过占位符 `_` 忽略元素
  - **dot product 示例对比**：C++17 需双层嵌套 `std::apply`；C++26 只需 `auto&& [...p_elems] = p; auto&& [...q_elems] = q;` 后直接 fold，零间接成本
  - 提案早期曾支持非模板上下文，因实现复杂度被搁置，但方向性明确
  - 对应提案为 P1061R10，与 C++26 的 pack indexing、constexpr bindings 等改进一起构成结构化绑定生态升级

### 2. best: 追求"更好"的 C++ STL 替换库 [🔗](https://github.com/mcy/best)
> **来源**：GitHub (mcy/best, 435 Stars)

- **一句话总结**：一个激进的 C++20 STL 替换库，对标 Abseil/Folly 但同时彻底替换标准库基础类型，设计哲学上大量借鉴 Rust。
- **核心要点**：
  - 目标不是兼容，而是「全面替换 40 年技术债务」——提供 `best::option`（类似 Rust Option）、`best::result`、`best::choice` 等词汇类型
  - 不友好异常，要求 C++20 起步，仅支持 Clang/GCC/Clang-Cl，使用 Bazel 构建
  - 模块化组织：`base/`（基础类型）、`container/`（容器）、`func/`（函数式支持）、`io/`（I/O）、`memory/`（内存管理）、`meta/`（元编程 & 反射）、`text/`（文本处理，Unicode-first）
  - API 风格混合：snake_case 命名（参考 STL），PascalCase 用于 FTADLE 扩展点，文档注释采用 Rust 风格
  - Apache 2.0 许可，部分算法复制自 Rust 的 libstd
  - 注意：项目仍处于早期构造阶段（36 commits），当前 CI 仅测试 x86_64 Linux + Clang，不建议生产依赖

### 3. When The C/C++ Users Journal Disappeared — 一段被遗忘的生态史 [🔗](https://freshsources.com/blog/files/cpp-source.html)
> **来源**：Fresh Sources (CUJ 前高级编辑 Chuck Allison 执笔)

- **一句话总结**：前 C/C++ Users Journal 高级编辑回顾了 C++ 生态最重要的出版平台从 BBS 时代到停刊再到 digital 传承的历史，是对现代 C++ 社区生态的珍贵反思。
- **核心要点**：
  - CUJ 前身是 1981 年 BDS C 用户组简讯 → C Users Journal → 1994 年更名为 C/C++ Users Journal
  - 2003 年底停刊后被 Dr. Dobb's 吸收，曾刊载 Bjarne Stroustrup、Herb Sutter、Scott Meyers、Andrei Alexandrescu 等众多元老的深度文章
  - 停刊后社区迅速响应，创立 The C++ Source（artima.com 托管），编委包括 Howard Hinnant、David Abrahams、Walter Bright、Jim Coplien、Kevlin Henney 等
  - Howard Hinnant 等人 2002 年就在 The C++ Source 上发表了关于 move semantics 的早期文章，远在 C++11 标准化之前
  - 文章是对「知识传播渠道」的致敬，也是对今天 Stack Overflow / AI 问答时代的对照反思
